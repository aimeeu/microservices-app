# Frontend Configuration Guide

This guide explains how the frontend application is configured to connect to the backend API using environment variables in Nomad job specifications.

## 📋 Overview

The frontend application uses a dynamic configuration approach that allows the backend API URL to be configured at deployment time through Nomad templates, without rebuilding the Docker image.

## 🏗️ Architecture

```
┌─────────────────────────────────────────────────────┐
│                 Nomad Job Deployment                │
│                                                     │
│  1. Nomad Template Engine                          │
│     ├─> Queries service discovery                  │
│     ├─> Gets healthy backend instances             │
│     └─> Generates config.js                        │
│                                                     │
│  2. Frontend Container                              │
│     ├─> Nginx serves config.js from /local/        │
│     ├─> Browser loads config.js                    │
│     └─> app.js reads window.APP_CONFIG             │
└─────────────────────────────────────────────────────┘
```

## 📁 Configuration Files

### 1. `config.js` (Default - in Docker image)

Default configuration for local development:

```javascript
window.APP_CONFIG = {
    BACKEND_API_URL: 'http://localhost:3000'
};
```

**Used by:**
- Docker Compose deployments
- Local development
- Fallback when Nomad template is not available

### 2. `config.template.js` (Template - in Docker image)

Template file showing the structure (not used at runtime):

```javascript
window.APP_CONFIG = {
    BACKEND_API_URL: '${BACKEND_API_URL}'
};
```

### 3. Generated `config.js` (Runtime - from Nomad template)

Generated by Nomad template at deployment time:

```javascript
window.APP_CONFIG = {
    BACKEND_API_URL: 'http://172.17.0.2:3000'
};
```

**Generated by:** Nomad template in job specification
**Location:** `/local/config.js` in allocation directory
**Served by:** Nginx from `/local/` directory

## 🔧 How It Works

### Step 1: Nomad Template Generation

In the Nomad job specification:

```hcl
template {
  data = <<EOH
// Backend API Configuration
// Generated by Nomad template at runtime
window.APP_CONFIG = {
{{- range nomadService "backend-api" }}
    BACKEND_API_URL: 'http://{{ .Address }}:{{ .Port }}'
{{- end }}
{{- if not (nomadService "backend-api") }}
    BACKEND_API_URL: 'http://localhost:3000'
{{- end }}
};
EOH
  destination = "local/config.js"
  change_mode = "restart"
}
```

**What happens:**
1. Nomad queries service discovery for `backend-api`
2. Gets IP address and port of healthy instances
3. Generates `config.js` with actual backend URL
4. Writes to `/local/config.js` in allocation directory
5. Restarts frontend if backend URL changes

### Step 2: Nginx Configuration

Nginx serves the generated config.js:

```nginx
location = /config.js {
    alias /local/config.js;
    add_header Content-Type application/javascript;
    add_header Cache-Control "no-cache, no-store, must-revalidate";
    expires 0;
}
```

**Key points:**
- Serves from `/local/config.js` (Nomad-generated)
- No caching (always fresh configuration)
- Overrides the default config.js in Docker image

### Step 3: Frontend Application

The frontend loads and uses the configuration:

```html
<!-- index.html -->
<script src="config.js"></script>
<script src="app.js"></script>
```

```javascript
// app.js
const API_URL = window.APP_CONFIG?.BACKEND_API_URL || 'http://localhost:3000';
console.log('Backend API URL:', API_URL);
```

**Fallback chain:**
1. Try `window.APP_CONFIG.BACKEND_API_URL` (from config.js)
2. Fall back to localhost if not available

## 🚀 Deployment Scenarios

### Scenario 1: Docker Compose (Local Development)

**Configuration:**
- Uses default `config.js` from Docker image
- Backend URL: `http://localhost:3000`

**No Nomad template needed**

```yaml
# docker-compose.yml
services:
  frontend:
    image: microservices-frontend:latest
    ports:
      - "8080:80"
```

### Scenario 2: Nomad with Service Discovery

**Configuration:**
- Nomad template generates `config.js`
- Backend URL: Discovered via Nomad service discovery
- Example: `http://172.17.0.2:3000`

**Nomad job specification:**

```hcl
task "frontend" {
  template {
    data = <<EOH
window.APP_CONFIG = {
{{- range nomadService "backend-api" }}
  {{- if .Healthy }}
    BACKEND_API_URL: 'http://{{ .Address }}:{{ .Port }}'
  {{- end }}
{{- end }}
};
EOH
    destination = "local/config.js"
    change_mode = "restart"
  }
}
```

### Scenario 3: Nomad with Static Configuration

**Configuration:**
- Nomad template with hardcoded URL
- Backend URL: Static value

**Nomad job specification:**

```hcl
task "frontend" {
  template {
    data = <<EOH
window.APP_CONFIG = {
    BACKEND_API_URL: 'http://backend.example.com:3000'
};
EOH
    destination = "local/config.js"
    change_mode = "restart"
  }
}
```

### Scenario 4: Nomad with Environment Variable

**Configuration:**
- Nomad template uses environment variable
- Backend URL: From Nomad variable or meta

**Nomad job specification:**

```hcl
task "frontend" {
  env {
    BACKEND_URL = "http://backend.example.com:3000"
  }
  
  template {
    data = <<EOH
window.APP_CONFIG = {
    BACKEND_API_URL: '{{ env "BACKEND_URL" }}'
};
EOH
    destination = "local/config.js"
    change_mode = "restart"
  }
}
```

## 🐛 Troubleshooting

### Issue: Frontend uses wrong backend URL

**Symptoms:**
- Frontend can't connect to backend
- Console shows incorrect URL

**Solutions:**

1. **Check generated config.js:**
   ```bash
   nomad alloc fs <alloc-id> local/config.js
   ```

2. **Verify Nomad template:**
   ```bash
   nomad job inspect microservices-app-local | jq '.Job.TaskGroups[].Tasks[].Templates'
   ```

3. **Check service discovery:**
   ```bash
   nomad service info backend-api
   ```

4. **Restart frontend allocation:**
   ```bash
   nomad alloc restart <alloc-id>
   ```

### Issue: config.js not found (404)

**Symptoms:**
- Browser console shows 404 for config.js
- Uses fallback URL

**Solutions:**

1. **Check Nginx configuration:**
   ```bash
   nomad alloc exec <alloc-id> cat /etc/nginx/conf.d/nginx.conf
   ```

2. **Verify file exists:**
   ```bash
   nomad alloc fs <alloc-id> local/config.js
   ```

3. **Check Nginx logs:**
   ```bash
   nomad alloc logs <alloc-id> frontend
   ```

### Issue: Configuration not updating

**Symptoms:**
- Backend URL doesn't change when backend moves
- Old configuration persists

**Solutions:**

1. **Check change_mode:**
   ```hcl
   template {
     change_mode = "restart"  # Should restart on change
   }
   ```

2. **Verify template is updating:**
   ```bash
   nomad alloc status <alloc-id>
   # Look for recent restarts
   ```

3. **Clear browser cache:**
   - Hard refresh: Ctrl+Shift+R (Windows/Linux) or Cmd+Shift+R (Mac)

## 💡 Best Practices

### 1. Always Provide Fallback

```javascript
const API_URL = window.APP_CONFIG?.BACKEND_API_URL || 'http://localhost:3000';
```

### 2. Log Configuration

```javascript
console.log('Backend API URL:', API_URL);
```

### 3. No Cache for config.js

```nginx
location = /config.js {
    add_header Cache-Control "no-cache, no-store, must-revalidate";
    expires 0;
}
```

### 4. Restart on Configuration Change

```hcl
template {
  change_mode = "restart"  # Restart when backend changes
}
```

### 5. Health Check Backend

```javascript
async function fetchStatus() {
    try {
        const response = await fetch(`${API_URL}/health`);
        // Handle response
    } catch (error) {
        console.error('Backend not available:', error);
    }
}
```

## 🎯 Summary

**Benefits of this approach:**

✅ **No image rebuild** - Change backend URL without rebuilding frontend
✅ **Dynamic discovery** - Automatically finds backend instances
✅ **Environment-specific** - Different URLs for dev/staging/prod
✅ **Zero downtime** - Updates configuration without manual intervention
✅ **Fallback support** - Works even if service discovery fails
✅ **Simple debugging** - Easy to verify configuration in browser

**Configuration flow:**

```
Nomad Template → config.js → Nginx → Browser → app.js → Backend API
